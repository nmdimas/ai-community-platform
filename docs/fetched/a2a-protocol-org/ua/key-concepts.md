# Основні концепції та компоненти A2A

A2A використовує набір основних концепцій, які визначають взаємодію між агентами. Розуміння цих базових будівельних блоків необхідне для розробки або інтеграції з системами, сумісними з A2A.

## Основні учасники взаємодії A2A

- **Користувач (User)**: Кінцевий користувач, яким може бути людина-оператор або автоматизований сервіс. Користувач ініціює запит або визначає мету, яка потребує допомоги одного чи кількох AI-агентів.
- **A2A Client (Client Agent)**: Застосунок, сервіс або інший AI-агент, що діє від імені користувача. Клієнт ініціює комунікацію за протоколом A2A.
- **A2A Server (Remote Agent)**: AI-агент або агентна система, що надає HTTP-ендпоінт з реалізацією протоколу A2A. Він отримує запити від клієнтів, обробляє задачі та повертає результати або оновлення статусу. З точки зору клієнта, віддалений агент працює як _непрозора_ (black-box) система, тобто його внутрішня логіка, пам'ять чи інструменти не розкриваються.

## Базові елементи комунікації

| Елемент | Опис | Основне призначення |
|---------|------|---------------------|
| Agent Card | JSON-документ з метаданими, що описує ідентичність агента, можливості, ендпоінт, навички та вимоги до автентифікації. | Дозволяє клієнтам виявляти агентів та розуміти, як безпечно та ефективно з ними взаємодіяти. |
| Task | Одиниця роботи зі станом, ініційована агентом, з унікальним ID та визначеним життєвим циклом. | Забезпечує відстеження тривалих операцій та підтримує багатокрокові взаємодії та співпрацю. |
| Message | Одиничний крок комунікації між клієнтом та агентом, що містить контент та роль ("user" або "agent"). | Передає інструкції, контекст, запитання, відповіді або оновлення статусу, які не обов'язково є формальними артефактами. |
| Part | Базовий контейнер контенту, що використовується в Message та Artifact. Part містить одне з: текстовий контент, посилання на файл (URL або вбудовані байти) або структуровані дані. | Забезпечує гнучкість для обміну різними типами контенту в повідомленнях та артефактах. |
| Artifact | Матеріальний результат, згенерований агентом під час виконання Task (наприклад, документ, зображення або структуровані дані). | Доставляє конкретні результати роботи агента, забезпечуючи структуровані та доступні для отримання виходи. |

## Механізми взаємодії

Протокол A2A підтримує різні патерни взаємодії:

- **Request/Response (Polling)**: Клієнт надсилає запит, сервер відповідає. Для тривалих задач клієнт періодично опитує сервер для отримання оновлень.
- **Стрімінг через Server-Sent Events (SSE)**: Клієнт ініціює потік для отримання результатів або оновлень статусу в реальному часі через відкрите HTTP-з'єднання.
- **Push-сповіщення**: Для дуже тривалих задач або сценаріїв з відключенням сервер може активно надсилати асинхронні сповіщення на webhook, наданий клієнтом, при значних оновленнях Task.

## Agent Cards

Agent Card — це JSON-документ, що слугує цифровою візиткою для початкового виявлення та налаштування взаємодії. Він надає важливі метадані про агента. Клієнти аналізують цю інформацію, щоб визначити, чи підходить агент для конкретної задачі, як структурувати запити та як безпечно комунікувати. Ключова інформація включає ідентичність, ендпоінт сервісу (URL), можливості A2A, вимоги до автентифікації та список навичок.

## Messages та Parts

Message представляє одиничний крок комунікації між клієнтом та агентом. Він включає роль ("user" або "agent") та унікальний `messageId`. Він містить один або більше об'єктів Part, які є гранулярними контейнерами для фактичного контенту.

Об'єкт `Part` — це гнучкий контейнер, що може містити різні типи контенту, використовуючи структуру поля `oneof`. Part повинен містити рівно одне з наступного:

- `text`: Рядок з текстовим контентом.
- `raw`: Масив байтів з бінарними даними файлу (вбудовано).
- `url`: Рядок URI з посиланням на зовнішній файловий контент.
- `data`: Структуроване JSON-значення (наприклад, об'єкт, масив) для машиночитаних даних.

Додатково кожен `Part` може включати:

- `mediaType`: MIME-тип контенту (наприклад, `"text/plain"`, `"image/png"`, `"application/json"`).
- `filename`: Необов'язкове ім'я файлу або контенту.
- `metadata`: Карта ключ-значення для додаткового контексту.

## Artifacts

Artifact представляє матеріальний результат або конкретний вихід, згенерований віддаленим агентом під час обробки Task. На відміну від загальних повідомлень, артефакти є фактичними результатами роботи. Artifact має унікальний `artifactId`, зрозумілу для людини назву та складається з одного або більше об'єктів Part. Артефакти тісно пов'язані з життєвим циклом Task і можуть стрімитися клієнту інкрементально.

## Відповідь агента: Task або Message

Відповідь агента може бути новим `Task` (коли агенту потрібно виконати тривалу операцію) або `Message` (коли агент може відповісти негайно).

## Інші важливі концепції

- **Контекст (`contextId`):** Ідентифікатор, згенерований сервером, який може використовуватися для логічного групування кількох пов'язаних об'єктів `Task`, забезпечуючи контекст для серії взаємодій.
- **Транспорт та формат:** Комунікація A2A відбувається через HTTP(S). JSON-RPC 2.0 використовується як формат корисного навантаження для всіх запитів та відповідей.
- **Автентифікація та авторизація:** A2A покладається на стандартні практики веб-безпеки. Вимоги до автентифікації оголошуються в Agent Card, а облікові дані (наприклад, OAuth-токени, API-ключі) зазвичай передаються через HTTP-заголовки, окремо від повідомлень протоколу A2A.
- **Agent Discovery:** Процес, за допомогою якого клієнти знаходять Agent Cards, щоб дізнатися про доступні A2A-сервери та їхні можливості.
- **Extensions:** A2A дозволяє агентам оголошувати кастомні розширення протоколу як частину їхнього AgentCard.
