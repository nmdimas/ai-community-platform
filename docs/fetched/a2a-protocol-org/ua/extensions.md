# Extensions в A2A

Extensions дозволяють розширювати протокол A2A новими даними, вимогами, RPC-методами та машинами станів. Агенти оголошують підтримку конкретних розширень у своєму Agent Card, а клієнти можуть обрати поведінку, яку пропонує розширення, в рамках запитів до агента.

## Сфера застосування Extensions

Extensions забезпечують кілька сценаріїв використання:

- **Data-only Extensions**: Надання нової структурованої інформації в Agent Card, що не впливає на потік запит-відповідь
- **Profile Extensions**: Накладання додаткової структури та вимог до зміни стану на основні повідомлення запит-відповідь
- **Method Extensions (Extended Skills)**: Додавання повністю нових RPC-методів поза основним набором
- **State Machine Extensions**: Додавання нових станів або переходів до машини станів завдань

## Перелік прикладів Extensions

| Extension | Опис |
|-----------|------|
| Secure Passport Extension | Додає надійний контекстний рівень для миттєвої персоналізації та зниження накладних витрат (v1) |
| Hello World or Timestamp Extension | Доповнює базові типи A2A, додаючи мітки часу до полів метаданих (v1) |
| Traceability Extension | Реалізація на Python та базове використання Traceability Extension (v1) |
| Agent Gateway Protocol (AGP) Extension | Впроваджує Autonomous Squads та маршрутизує Intent payloads на основі оголошених Capabilities (v1) |

## Обмеження

Extensions не можуть:

- Змінювати визначення основних структур даних
- Додавати нові значення до enum-типів

Extensions повинні розміщувати користувацькі атрибути в map `metadata`, присутньому в основних структурах даних.

## Оголошення Extension

Агенти оголошують підтримку розширень у своєму Agent Card, включаючи об'єкти `AgentExtension` до об'єкта `AgentCapabilities`.

| Поле | Тип | Обов'язкове | Опис |
|------|-----|-------------|------|
| `uri` | string | Ні | Унікальний URI, що ідентифікує розширення |
| `description` | string | Ні | Зрозумілий для людини опис того, як агент використовує розширення |
| `required` | boolean | Ні | Якщо true, клієнт повинен розуміти та дотримуватися вимог розширення |
| `params` | object | Ні | Параметри конфігурації, специфічні для розширення |

### Приклад Agent Card з Extension

```json
{
  "name": "Magic 8-ball",
  "description": "An agent that can tell your future... maybe.",
  "version": "0.1.0",
  "url": "https://example.com/agents/eightball",
  "capabilities": {
    "streaming": true,
    "extensions": [
      {
        "uri": "https://example.com/ext/konami-code/v1",
        "description": "Provide cheat codes to unlock new fortunes",
        "required": false,
        "params": {
          "hints": [
            "When your sims need extra cash fast",
            "You might deny it, but we've seen the evidence of those cows."
          ]
        }
      }
    ]
  },
  "defaultInputModes": ["text/plain"],
  "defaultOutputModes": ["text/plain"],
  "skills": [
    {
      "id": "fortune",
      "name": "Fortune teller",
      "description": "Seek advice from the mystical magic 8-ball",
      "tags": ["mystical", "untrustworthy"]
    }
  ]
}
```

## Обов'язкові Extensions

Коли Agent Card оголошує розширення як `required: true`, це сигналізує клієнтам, що деякий аспект розширення впливає на те, як структуруються або обробляються запити. Клієнти повинні дотримуватися обов'язкових розширень. Агентам не слід позначати data-only extensions як обов'язкові.

## Специфікація Extension

Детальна поведінка розширення визначається його специфікацією, яка повинна містити принаймні:

- Конкретний URI (або URI), що ідентифікують розширення
- Схему та значення об'єктів у полі `params`
- Схеми додаткових структур даних, якими обмінюються клієнт та агент
- Деталі нових потоків запит-відповідь або будь-якої іншої необхідної логіки

## Залежності Extensions

Extensions можуть залежати від інших розширень -- як обов'язкових, так і необов'язкових. Специфікації розширень повинні документувати ці залежності. Активація розширення та всіх його обов'язкових залежностей є відповідальністю клієнта.

## Активація Extension

За замовчуванням розширення неактивні. Клієнти та агенти виконують узгодження, щоб визначити, які розширення активні для конкретного запиту.

1. **Запит клієнта**: Клієнт запитує активацію розширення, включаючи заголовок `A2A-Extensions` у HTTP-запит
2. **Обробка агентом**: Агенти ідентифікують підтримувані розширення у запиті та виконують активацію
3. **Відповідь**: Відповідь агента ПОВИННА включати заголовок `A2A-Extensions` зі списком усіх успішно активованих розширень

### Приклад запиту

```
POST /agents/eightball HTTP/1.1
Host: example.com
Content-Type: application/json
A2A-Extensions: https://example.com/ext/konami-code/v1
Content-Length: 519

{
  "jsonrpc": "2.0",
  "method": "SendMessage",
  "id": "1",
  "params": {
    "message": {
      "messageId": "1",
      "role": "ROLE_USER",
      "parts": [{"text": "Oh magic 8-ball, will it rain today?"}]
    },
    "metadata": {
      "https://example.com/ext/konami-code/v1/code": "motherlode"
    }
  }
}
```

### Відповідна відповідь

```
HTTP/1.1 200 OK
Content-Type: application/json
A2A-Extensions: https://example.com/ext/konami-code/v1
Content-Length: 338

{
  "jsonrpc": "2.0",
  "id": "1",
  "result": {
    "message": {
      "messageId": "2",
      "role": "ROLE_AGENT",
      "parts": [{"text": "That's a bingo!"}]
    }
  }
}
```

## Зауваження щодо реалізації

### Версіонування

- Використовуйте URI розширення як основний ідентифікатор версії (наприклад, `https://example.com/ext/my-extension/v1`)
- Новий URI ПОВИНЕН використовуватися при введенні несумісних змін
- Якщо клієнт запитує непідтримувану версію, агент ПОВИНЕН ігнорувати запит на активацію

### Виявлення та публікація

- Документ специфікації розширення повинен бути розміщений за URI розширення
- Авторам рекомендується використовувати сервіси постійних ідентифікаторів, такі як `w3id.org`

### Пакування та повторне використання

Логіку розширень слід пакувати в повторно використовувані бібліотеки:

- Розповсюджувати як стандартний пакет для відповідної мовної екосистеми (PyPI для Python, npm для TypeScript/JavaScript)
- Забезпечувати спрощену інтеграцію з мінімальним обсягом необхідного коду

### Безпека

- **Валідація вхідних даних**: Усі нові поля даних, параметри або методи повинні ретельно валідуватися
- **Область дії обов'язкових розширень**: Будьте обережні при позначенні розширень як `required: true`
- **Автентифікація та авторизація**: Нові методи розширень повинні підлягати тим самим перевіркам, що й основні методи A2A. Extensions НЕ ПОВИННІ надавати спосіб обійти основні засоби безпеки агента
