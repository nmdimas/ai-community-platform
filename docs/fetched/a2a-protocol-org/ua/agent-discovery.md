# Agent Discovery в A2A

Для співпраці за протоколом Agent2Agent (A2A) AI-агенти спочатку повинні знайти один одного та зрозуміти свої можливості. A2A стандартизує самоопис агентів через **Agent Card**. Однак методи виявлення цих Agent Cards варіюються залежно від середовища та вимог. Agent Card визначає, що пропонує агент. Існують різні стратегії для клієнтського агента щодо виявлення цих карток. Вибір стратегії залежить від середовища розгортання та вимог безпеки.

## Роль Agent Card

Agent Card — це JSON-документ, що слугує цифровою "візиткою" для A2A Server (віддаленого агента). Він є критично важливим для виявлення агентів та взаємодії з ними. Ключова інформація, що міститься в Agent Card:

- **Ідентичність:** Включає `name`, `description` та інформацію про `provider`.
- **Ендпоінт сервісу:** Вказує `url` для сервісу A2A.
- **Можливості A2A:** Перелік підтримуваних функцій, таких як `streaming` або `pushNotifications`.
- **Автентифікація:** Деталі необхідних `schemes` (наприклад, "Bearer", "OAuth2").
- **Навички:** Описує задачі агента за допомогою об'єктів `AgentSkill`, включаючи `id`, `name`, `description`, `inputModes`, `outputModes` та `examples`.

Клієнтські агенти використовують Agent Card для визначення відповідності агента, структурування запитів та забезпечення безпечної комунікації.

## Стратегії виявлення

### 1. Well-Known URI

Цей підхід рекомендований для публічних агентів або агентів, призначених для широкого виявлення в межах конкретного домену.

- **Механізм:** A2A-сервери роблять свій Agent Card доступним для виявлення, розміщуючи його за стандартизованим `well-known` URI на своєму домені. Стандартний шлях — `https://{agent-server-domain}/.well-known/agent-card.json`, відповідно до принципів RFC 8615.

- **Процес:**

  1. Клієнтський агент знає або програмно виявляє домен потенційного A2A-сервера (наприклад, `smart-thermostat.example.com`).
  2. Клієнт виконує HTTP GET запит до `https://smart-thermostat.example.com/.well-known/agent-card.json`.
  3. Якщо Agent Card існує та доступний, сервер повертає його як JSON-відповідь.

- **Переваги:** Простота реалізації, відповідність стандартам, сприяння автоматизованому виявленню.

- **Зауваження:** Найкраще підходить для відкритих сценаріїв або виявлення в межах контрольованого домену. Автентифікація необхідна на ендпоінті, що обслуговує Agent Card, якщо він містить конфіденційні дані.

### 2. Куратовані реєстри (виявлення через каталог)

Цей підхід застосовується в ентерпрайз-середовищах або публічних маркетплейсах, де Agent Cards часто управляються центральним реєстром.

- **Механізм:** Проміжний сервіс (реєстр) зберігає колекцію Agent Cards. Клієнти запитують цей реєстр для пошуку агентів за різними критеріями (наприклад, пропоновані навички, теги, ім'я провайдера, можливості).

- **Процес:**

  1. A2A-сервери публікують свої Agent Cards у реєстрі.
  2. Клієнтські агенти запитують API реєстру та шукають за критеріями, такими як "конкретні навички".
  3. Реєстр повертає відповідні Agent Cards або посилання на них.

- **Переваги:** Централізоване управління та governance, виявлення за можливостями, підтримка контролю доступу та фреймворків довіри, застосовність як у приватних, так і в публічних маркетплейсах.

- **Зауваження:** Потребує розгортання та підтримки сервісу реєстру. Поточна специфікація A2A не визначає стандартний API для курованих реєстрів.

### 3. Пряма конфігурація / Приватне виявлення

Цей підхід використовується для тісно зв'язаних систем, приватних агентів або з метою розробки.

- **Механізм:** Клієнтські застосунки використовують жорстко закодовані дані, файли конфігурації, змінні середовища або пропрієтарні API для виявлення.
- **Процес:** Процес залежить від стратегії розгортання та конфігурації застосунку.
- **Переваги:** Простий спосіб встановлення з'єднань у відомих, статичних відносинах.
- **Зауваження:** Негнучкий для сценаріїв динамічного виявлення. Зміни в інформації Agent Card потребують переконфігурації клієнта.

## Захист Agent Cards

Agent Cards містять конфіденційну інформацію, таку як URL внутрішніх або обмежених агентів та описи чутливих навичок.

### Механізми захисту

- **Автентифіковані Agent Cards:** Використовуйте автентифіковані розширені Agent Cards для конфіденційної інформації або для надання більш детальної версії картки.
- **Захищені ендпоінти:** Реалізуйте контроль доступу на HTTP-ендпоінті, що обслуговує Agent Card. Методи включають:
  - Mutual TLS (mTLS)
  - Мережеві обмеження (наприклад, діапазони IP)
  - HTTP-автентифікація (наприклад, OAuth 2.0)
- **Вибіркове розкриття реєстром:** Реєстри повертають різні Agent Cards залежно від ідентичності та дозволів клієнта.

Будь-який Agent Card, що містить конфіденційні дані, повинен бути захищений механізмами автентифікації та авторизації. Специфікація A2A рекомендує використовувати позасмугові динамічні облікові дані замість вбудовування статичних секретів в Agent Card.

## Перспективи розвитку

Спільнота A2A досліджує стандартизацію взаємодії з реєстрами та просунуті протоколи виявлення.
