# Streaming та асинхронні операції

Протокол Agent2Agent (A2A) обробляє завдання, які можуть не завершитися миттєво. Багато операцій, керованих ШІ, є тривалими, включають декілька кроків, дають проміжні результати або потребують втручання людини. A2A надає механізми для управління такими асинхронними взаємодіями, забезпечуючи ефективне отримання оновлень клієнтами як при постійному з'єднанні, так і в режимі без з'єднання.

## Streaming за допомогою Server-Sent Events (SSE)

Для завдань, що дають проміжні результати або постійні оновлення статусу, A2A підтримує комунікацію в реальному часі з використанням Server-Sent Events (SSE).

### Ключові можливості

- **Можливості сервера:** A2A Server вказує на підтримку streaming, встановлюючи `capabilities.streaming: true` у своєму Agent Card.
- **Ініціювання потоку:** Клієнти використовують RPC-метод `SendStreamingMessage` для надсилання початкового повідомлення та одночасної підписки на оновлення завдання.
- **Відповідь сервера:** При успішній підписці сервер відповідає HTTP 200 OK та `Content-Type: text/event-stream`, тримаючи з'єднання відкритим для надсилання подій.
- **Структура подій:** Сервер надсилає події, що містять об'єкти JSON-RPC 2.0 Response, зазвичай `SendStreamingMessageResponse`. Поле result включає:
  - `Task`: Поточний стан роботи
  - `TaskStatusUpdateEvent`: Зміни стану життєвого циклу та проміжні повідомлення
  - `TaskArtifactUpdateEvent`: Нові або оновлені артефакти, що передаються частинами
- **Завершення потоку:** Коли завдання досягають термінального або перерваного стану (COMPLETED, FAILED, CANCELED, REJECTED, INPUT_REQUIRED), сервер закриває потік.
- **Повторна підписка:** Клієнти можуть перепідключитися за допомогою RPC-методу `SubscribeToTask`, якщо з'єднання передчасно розірвалося.

### Коли використовувати Streaming

- Моніторинг прогресу тривалих завдань у реальному часі
- Отримання великих результатів поступово
- Інтерактивні обміни, що потребують негайного зворотного зв'язку
- Застосунки, що потребують оновлень від агентів з низькою затримкою

## Push-сповіщення для сценаріїв без з'єднання

Для дуже тривалих завдань або коли клієнти не можуть підтримувати постійне з'єднання, A2A підтримує асинхронні оновлення через push-сповіщення.

### Ключові можливості

- **Можливості сервера:** Сервери вказують на підтримку, встановлюючи `capabilities.pushNotifications: true` у своєму Agent Card.
- **Конфігурація:** Клієнти надають `PushNotificationConfig` серверу або в початкових запитах `SendMessage`/`SendStreamingMessage`, або окремо за допомогою `CreateTaskPushNotificationConfig`. Конфігурація включає URL вебхука, необов'язковий токен та необов'язкові дані автентифікації.
- **Тригер сповіщення:** Сервер надсилає сповіщення при значних змінах стану.
- **Тіло сповіщення:** HTTP-тіло містить об'єкти `StreamResponse`, що включають `task`, `message`, `statusUpdate` або `artifactUpdate`.
- **Дія клієнта:** Після отримання перевірених сповіщень клієнти зазвичай використовують RPC-метод `GetTask` для отримання повного оновленого об'єкта `Task`.

### Коли використовувати push-сповіщення

- Дуже тривалі завдання, що тривають хвилини, години або дні
- Клієнти, нездатні підтримувати постійне з'єднання (мобільні застосунки, serverless-функції)
- Сценарії, що потребують сповіщення лише при значних змінах стану

## Клієнтський сервіс push-сповіщень

`PushNotificationConfig.url` вказує на клієнтський сервіс Push-сповіщень, відповідальний за отримання HTTP POST-сповіщень від A2A Server. Цей сервіс автентифікує вхідні сповіщення, перевіряє їх релевантність та передає вміст відповідній логіці клієнтського застосунку.

## Зауваження щодо безпеки push-сповіщень

### Безпека A2A Server

**Валідація URL вебхука:** Сервери не повинні сліпо довіряти URL, наданим клієнтом, щоб запобігти SSRF або DDoS-атакам. Стратегії пом'якшення включають список дозволених доменів, перевірку власності та контроль вихідного мережевого трафіку.

**Автентифікація до вебхуків клієнта:** A2A Server повинен автентифікуватися відповідно до схеми в `PushNotificationConfig.authentication` (Bearer Tokens, API keys, HMAC-підписи або mutual TLS).

### Безпека клієнтського отримувача вебхуків

**Автентифікація A2A Server:** Перевіряйте автентичність вхідних сповіщень за допомогою верифікації підпису/токена та необов'язкового `PushNotificationConfig.token`.

**Запобігання replay-атакам:**
- Включайте мітки часу та відхиляйте занадто старі сповіщення
- Використовуйте унікальні ідентифікатори (JWT `jti` claims, event IDs) для критичних сповіщень

**Управління ключами:** Впроваджуйте безпечні практики, включаючи регулярну ротацію ключів. Протоколи, такі як JWKS, полегшують ротацію асиметричних ключів.

### Приклад потоку з асиметричними ключами (JWT + JWKS)

1. Клієнт створює `PushNotificationConfig` з `authentication.scheme: "Bearer"` та очікуваними issuer/audience для JWT.
2. A2A Server генерує JWT, підписаний своїм приватним ключем, включаючи claims (`iss`, `aud`, `iat`, `exp`, `jti`, `taskId`), та робить публічні ключі доступними через JWKS endpoint.
3. Клієнтський вебхук перевіряє підпис JWT за допомогою публічного ключа з JWKS endpoint, валідує claims та перевіряє необов'язковий токен.
