# Ентерпрайз-впровадження A2A

Протокол Agent2Agent (A2A) спроєктований з урахуванням ентерпрайз-вимог у своїй основі. Замість винаходу нових пропрієтарних стандартів для безпеки та операцій, A2A прагне безперешкодно інтегруватися з існуючою ентерпрайз-інфраструктурою та широко прийнятими найкращими практиками. Такий підхід дозволяє організаціям використовувати наявні інвестиції та експертизу в безпеці, моніторингу, governance та управлінні ідентичностями.

Ключовий принцип A2A полягає в тому, що агенти зазвичай є **непрозорими**, оскільки вони не діляться внутрішньою пам'яттю, інструментами чи прямим доступом до ресурсів один з одним. Ця непрозорість природно узгоджується зі стандартними парадигмами безпеки клієнт-сервер, трактуючи віддалених агентів як стандартні HTTP-базовані ентерпрайз-застосунки.

## Transport Level Security (TLS)

- **Обов'язковість HTTPS**: Вся комунікація A2A у продакшн-середовищах повинна відбуватися через `HTTPS`.
- **Сучасні стандарти TLS**: Реалізації повинні використовувати сучасні версії TLS. Рекомендується TLS 1.2 або вище.
- **Верифікація ідентичності сервера**: A2A-клієнти повинні верифікувати ідентичність A2A-сервера шляхом перевірки його TLS-сертифіката відносно довірених центрів сертифікації під час TLS handshake.

## Автентифікація

A2A делегує автентифікацію стандартним веб-механізмам. Протокол переважно покладається на HTTP-заголовки та усталені стандарти, такі як OAuth2 та OpenID Connect.

- **Відсутність ідентичності в корисному навантаженні**: Корисне навантаження протоколу A2A не містить інформації про ідентичність користувача чи клієнта напряму. Ідентичність встановлюється на рівні транспорту/HTTP.
- **Оголошення в Agent Card**: Agent Card A2A-сервера описує підтримувані схеми автентифікації у полі `security`.
- **Позасмугове отримання облікових даних**: A2A-клієнт отримує необхідні облікові дані через процеси, зовнішні відносно самого протоколу A2A.
- **Передача через HTTP-заголовки**: Облікові дані **повинні** передаватися у стандартних HTTP-заголовках (наприклад, `Authorization: Bearer <TOKEN>`).
- **Серверна валідація**: A2A-сервер **повинен** автентифікувати кожен вхідний запит.
  - `401 Unauthorized`: Облікові дані відсутні або недійсні.
  - `403 Forbidden`: Облікові дані дійсні, але недостатні дозволи.
- **Автентифікація під час Task (вторинні облікові дані)**: Якщо агенту потрібні додаткові облікові дані під час виконання Task, A2A-сервер повідомляє клієнту про необхідність надання додаткової інформації.

## Авторизація

- **Гранулярний контроль**: Авторизація **повинна** застосовуватися на основі автентифікованої ідентичності.
- **Авторизація на рівні навичок**: Доступ може контролюватися для кожної навички окремо, як оголошено в Agent Card.
- **Авторизація на рівні даних та дій**: Агенти **повинні** забезпечувати належну авторизацію перед виконанням чутливих дій.
- **Принцип найменших привілеїв**: Агенти **повинні** надавати лише необхідні дозволи.

## Конфіденційність та захист даних

- **Усвідомлення чутливості**: Розробники повинні бути обізнані про чутливість даних, що обмінюються в частинах Message та Artifact.
- **Відповідність нормативам**: Забезпечення відповідності відповідним нормативним актам щодо конфіденційності даних (GDPR, CCPA, HIPAA).
- **Мінімізація даних**: Уникайте включення надмірно чутливої інформації в обміни A2A.
- **Безпечна обробка**: Захист даних як під час передачі (TLS), так і в стані спокою.

## Трейсинг, спостережуваність та моніторинг

- **Розподілений трейсинг**: Використовуйте OpenTelemetry для передачі контексту трейсів через стандартні HTTP-заголовки (W3C Trace Context).
- **Комплексне логування**: Логуйте taskId, sessionId, кореляційні ID та контекст трейсів.
- **Метрики**: Надавайте ключові операційні метрики (частота запитів, частота помилок, латентність, використання ресурсів).
- **Аудит**: Проводьте аудит значущих подій, особливо тих, що стосуються чутливих даних або операцій з високим впливом.

## Управління API та governance

Для A2A-серверів, що надаються зовнішнім споживачам або в межах великих підприємств, інтеграція з рішеннями для управління API забезпечує:

- **Централізоване застосування політик**: Автентифікація, авторизація, обмеження частоти запитів, квоти.
- **Управління трафіком**: Балансування навантаження, маршрутизація, медіація.
- **Аналітика та звітність**: Інсайти щодо використання, продуктивності та тенденцій.
- **Портали для розробників**: Виявлення, документація (Agent Cards) та онбординг.
