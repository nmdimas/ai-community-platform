# Життєвий цикл Task

## Огляд

У протоколі Agent2Agent (A2A) взаємодії можуть бути як простими та без стану, так і складними та тривалими. Агенти відповідають двома основними способами:

- **`Message` без стану**: Для миттєвих, самодостатніх взаємодій
- **`Task` зі станом**: Для обробки через визначений життєвий цикл до досягнення перерваного або термінального стану

## Групування пов'язаних взаємодій

`contextId` логічно групує декілька об'єктів `Task` та `Message`, забезпечуючи безперервність між взаємодіями.

- Перше повідомлення генерує новий `contextId` та, за бажанням, `taskId`
- Наступні повідомлення включають той самий `contextId` для продовження взаємодії
- Клієнти за бажанням додають `taskId`, щоб вказати на продовження конкретного завдання

`contextId` забезпечує співпрацю для досягнення спільних цілей, одночасно керуючи внутрішнім станом розмови.

## Відповідь агента: Message або Task

Вибір залежить від характеру взаємодії та можливостей агента:

- **Message для простих взаємодій**: Підходить для транзакційних обмінів без тривалої обробки
- **Task для взаємодій зі станом**: Використовується, коли відображення наміру на можливості потребує значної, відстежуваної роботи

### Типи агентів

- **Агенти лише з Message**: Завжди відповідають об'єктами `Message`; використовують `contextId` для безперервності
- **Агенти, що генерують Task**: Завжди відповідають об'єктами `Task`, уникаючи рішення між Message та Task
- **Гібридні агенти**: Генерують обидва типи; використовують повідомлення для узгодження обсягу, потім створюють завдання для відстеження виконання

## Уточнення Task

Клієнти уточнюють результати, надсилаючи нові запити з тим самим `contextId`, що й у оригінального завдання. Клієнти надають `referenceTaskIds` в об'єктах `Message`, а агенти відповідають новим `Task` або `Message`.

## Незмінність Task

Термінальні стани (completed, canceled, rejected, failed) є постійними. Подальші взаємодії повинні ініціювати нові завдання в межах того самого `contextId`.

**Переваги:**
- Надійні посилання на завдання та чітке відображення вхідних/вихідних даних
- Чіткі одиниці роботи для детального відстеження
- Спрощена розробка агентів завдяки усуненню неоднозначності перезапуску

## Паралельні продовження

A2A підтримує паралельну роботу через окремі завдання в межах одного `contextId`, що дозволяє відстежувати залежності та послідовне виконання після завершення передумов.

Приклад робочого процесу:
- Task 1: Забронювати авіаквиток
- Task 2: Забронювати готель (залежить від Task 1)
- Task 3: Забронювати активність (залежить від Task 1)
- Task 4: Додати бронювання спа (залежить від Task 2)

## Посилання на попередні Artifact

Обслуговуючі агенти виводять відповідні артефакти з посилань на завдання або `contextId`. Якщо є неоднозначність, агенти повертають стан `input-required` із запитом на уточнення.

## Відстеження змін Artifact

Клієнти керують зв'язуванням артефактів та історією версій, а не обслуговуючий агент. Агенти повинні використовувати однакове `artifact-name` при створенні уточнених версій.

**Обов'язки клієнта:**
- Явно посилатися на конкретні артефакти для уточнення
- Підтримувати історію версій
- Показувати користувачам останню прийнятну версію

## Приклад сценарію з продовженням

**Крок 1:** Клієнт запитує генерацію зображення
```json
{
  "jsonrpc": "2.0",
  "id": "req-001",
  "method": "SendMessage",
  "params": {
    "message": {
      "role": "user",
      "parts": [{ "text": "Generate an image of a sailboat on the ocean." }],
      "messageId": "msg-user-001"
    }
  }
}
```

**Крок 2:** Агент відповідає завершеним завданням із зображенням човна
```json
{
  "jsonrpc": "2.0",
  "id": "req-001",
  "result": {
    "task": {
      "id": "task-boat-gen-123",
      "contextId": "ctx-conversation-abc",
      "status": { "state": "TASK_STATE_COMPLETED" },
      "artifacts": [{
        "artifactId": "artifact-boat-v1-xyz",
        "name": "sailboat_image.png",
        "description": "A generated image of a sailboat on the ocean.",
        "parts": [{
          "filename": "sailboat_image.png",
          "mediaType": "image/png",
          "raw": "base64_encoded_png_data_of_a_sailboat"
        }]
      }]
    }
  }
}
```

**Крок 3:** Клієнт запитує уточнення, посилаючись на оригінальне завдання
```json
{
  "jsonrpc": "2.0",
  "id": "req-002",
  "method": "SendMessage",
  "params": {
    "message": {
      "role": "user",
      "messageId": "msg-user-002",
      "contextId": "ctx-conversation-abc",
      "referenceTaskIds": ["task-boat-gen-123"],
      "parts": [{ "text": "Please modify the sailboat to be red." }]
    }
  }
}
```

**Крок 4:** Агент відповідає новим завданням, у тому ж контексті, з тим самим ім'ям артефакту
```json
{
  "jsonrpc": "2.0",
  "id": "req-002",
  "result": {
    "task": {
      "id": "task-boat-color-456",
      "contextId": "ctx-conversation-abc",
      "status": { "state": "TASK_STATE_COMPLETED" },
      "artifacts": [{
        "artifactId": "artifact-boat-v2-red-pqr",
        "name": "sailboat_image.png",
        "description": "A generated image of a red sailboat on the ocean.",
        "parts": [{
          "filename": "sailboat_image.png",
          "mediaType": "image/png",
          "raw": "base64_encoded_png_data_of_a_RED_sailboat"
        }]
      }]
    }
  }
}
```
