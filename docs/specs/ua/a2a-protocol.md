# A2A-Протокол Для Агентів

## Мета

`A2A` (`agent-to-agent`) є внутрішнім контрактом взаємодії між core-agent, спеціалізованими агентами та іншими platform-модулями, які виконують автономні задачі.

## Призначення

A2A потрібен для того, щоб:

- core-agent міг делегувати задачі іншим агентам
- відповіді були уніфікованими незалежно від реалізації конкретного агента
- clarification loop і multi-step orchestration працювали через один контракт

## Базові Вимоги

- A2A повинен бути уніфікованим протоколом для всіх агентних викликів
- кожен агент повинен мати зрозумілу discovery-точку або registry-опис
- request і response повинні мати стабільну структуру
- кореляція викликів повинна бути явною (`request_id`, `trace_id`, за потреби `conversation_id`)

## Мінімальна Модель Взаємодії

- `request` містить контекст, намір, payload і metadata
- `response` повертає або результат, або потребу в уточненні, або помилку
- агент не повинен повертати неструктурований довільний payload без статусу

## Базові Статуси Відповіді

- `completed` — агент успішно завершив задачу
- `needs_clarification` — агенту потрібне уточнення
- `failed` — агент не зміг коректно виконати задачу
- `queued` — опційний статус для асинхронних сценаріїв майбутніх версій

## Clarification Loop

- якщо агент повертає `needs_clarification`, він повинен явно вказати, чого бракує
- core-agent або orchestrator повинен мати змогу передати follow-up answer у тому ж логічному ланцюжку
- clarification не повинно розривати кореляцію запиту

## Вимоги До Payload

- payload повинен бути структурованим
- джерела, докази або evidence повинні передаватись окремими полями, а не лише текстом
- agent-specific поля допустимі, якщо вони не ламають базову обгортку контракту

## Таймаути, Повтори, Ідемпотентність

- повторний виклик із тим самим `request_id` не повинен створювати непередбачуваного дублювання
- таймаути повинні бути явною частиною контракту виклику
- retry-поведінка повинна бути контрольованою і без "шторму" повторів

## Вимоги До Агентів

- кожен агент повинен або підтримувати A2A, або бути адаптованим через platform-owned wrapper
- агент не повинен ламати базовий envelope контракт
- помилки мають повертатися структуровано, а не "мовчки"

## Out Of Scope Для MVP

- складний streaming-протокол
- міжкластерна федерація агентів
- довільні несумісні agent-specific transport-моделі
