# Вимоги До Admin-Шару

## Мета

`Admin`-шар є внутрішнім інтерфейсом для керування платформою, агентами, конфігураціями та moderation-процесами.

## Основні Вимоги

- admin повинен бути логічно й технічно відокремлений від public web
- admin повинен працювати тільки для авторизованих користувачів із відповідними ролями
- усі критичні platform-операції повинні бути доступні через admin або через контрольовані chat-команди

## Обов'язкові Admin-Сценарії

- перегляд списку агентів і їх статусів
- ввімкнення / вимкнення агентів
- перегляд конфігів агентів
- перегляд moderation-черги, якщо вона присутня
- перегляд технічного стану інтеграцій і фонових задач

## Доступ І Ролі

- `admin` має повний доступ до platform-керування
- `moderator` може мати обмежений доступ до moderation і knowledge-related операцій
- `user` не повинен мати доступу до admin-шару

## Аудит І Контроль

- усі важливі admin-дії повинні логуватись
- зміни конфігів агентів повинні бути відстежуваними
- небезпечні дії не повинні виконуватись без явного контексту ролі

## Вимоги До Агентів

- агенти не повинні напряму обходити admin-рішення
- якщо агентна поведінка змінюється конфігом, цей конфіг повинен бути керований platform-owned шляхом
- agent state не повинен жити тільки в UI без platform persistence

## Безпека

- admin-шар повинен бути захищений аутентифікацією та авторизацією
- admin routes не повинні бути доступні як public endpoints без перевірок
- доступ до admin API має бути обмежений role-aware правилами

## Out Of Scope Для MVP

- складні кастомні workflow-конструктори
- granular RBAC нижче базових ролей
- окремий багаторівневий approval engine
