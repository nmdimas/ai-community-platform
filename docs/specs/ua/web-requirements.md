# Вимоги До Web-Шару

## Мета

`Web`-шар є public-facing інтерфейсом платформи. Він повинен надавати зрозумілий доступ до даних ком'юніті без необхідності працювати напряму з чат-командами або admin-інтерфейсом.

## Основні Вимоги

- public web повинен бути доступний окремо від admin-шару
- web повинен працювати без доступу до внутрішніх admin-функцій
- web повинен читати дані лише через platform-owned API або внутрішні application services
- web має підтримувати стабільні URL для основних публічних модулів

## Базові Public Surface

- сторінка каталогу локацій
- список / детальна сторінка knowledge-матеріалів
- публічні digest-представлення, якщо вони ввімкнені

## URL І Навігація

- public web не повинен використовувати admin-prefixed маршрути
- публічні маршрути мають бути передбачуваними, короткими і стабільними
- deep links з чату повинні вести в public web без додаткового admin-контексту

## Безпека І Доступ

- public web за замовчуванням не вимагає admin-аутентифікації
- операції запису через public web у MVP або заборонені, або жорстко обмежені
- будь-який доступ до приватних або moderation-даних через web заборонений

## Вимоги До Даних

- web не є source of truth для даних
- усі зміни даних відбуваються через core-platform
- web повинен показувати тільки дозволений publishable state

## Вимоги До Агентів

- агенти можуть формувати deep links у public web
- агенти не повинні покладатися на нестабільні або тимчасові UI-маршрути
- якщо агент посилається на web, URL має бути стабільною частиною контракту

## Out Of Scope Для MVP

- складний персоналізований кабінет користувача
- self-service admin-функції у public web
- складна client-side permission model
