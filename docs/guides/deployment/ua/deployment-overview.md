# Огляд деплою

Brama Agent Platform підтримує два офіційних режими деплою:

| Режим | Найкраще для | Пакування |
|-------|-------------|-----------|
| **Docker** | Хобі-продакшн, одновузлові самостійні інсталяції, локальна розробка | Docker Compose + Makefile |
| **Kubernetes** | Cluster-native оператори, керовані бази даних, rolling upgrades | Helm chart (заплановано) |

Обидва режими поділяють один [контракт деплою](./docker-deployment-contract.md): однакові
конфігураційні входи, очікування щодо здоров'я, поведінку міграцій та ворота верифікації оновлень.

Англійська версія: [deployment-overview.md](../en/deployment-overview.md)

---

## Підтримувані топології

### Docker (підтримується)

```
Один хост
  ├── Traefik (edge-проксі)
  ├── Core-платформа + планувальник
  ├── LiteLLM (LLM-проксі)
  ├── PostgreSQL, Redis, OpenSearch, RabbitMQ (вбудовані)
  └── Опціонально: агенти, Langfuse, OpenClaw
```

Всі сервіси запускаються на одному хості. Stateful-сервіси вбудовані як Docker-томи.

**Гайди для оператора:**
- [Встановлення](./docker-install.md)
- [Оновлення](./docker-upgrade.md)
- [Резервне копіювання та відновлення](./docker-backup-restore.md)
- [Вирішення проблем](./docker-troubleshooting.md)
- [Контракт деплою](./docker-deployment-contract.md)

### Kubernetes (заплановано)

```
Кластер
  ├── Core-платформа (Deployment, 1+ реплік)
  ├── Core-планувальник (Deployment, 1 репліка)
  ├── LiteLLM (Deployment)
  ├── Агенти (Deployments)
  ├── Ingress (Traefik або nginx-ingress)
  ├── PostgreSQL (вбудований або зовнішній керований)
  ├── Redis (вбудований або зовнішній керований)
  └── Опціонально: Langfuse, OpenClaw
```

Stateful-сервіси можуть бути замінені зовнішніми керованими сервісами (RDS, ElastiCache тощо).

## Порівняння

| Аспект | Docker | Kubernetes |
|--------|--------|------------|
| Складність налаштування | Низька | Висока |
| Операційне навантаження | Низьке | Середнє–Високе |
| Масштабування | Один хост | Мульти-репліка, горизонтальне |
| Stateful-сервіси | Вбудовані томи | Вбудовані або зовнішні керовані |
| Механізм оновлення | `git pull` + `make up` | `helm upgrade` |
| Відкат | `git checkout` + `make up` | `helm rollback` |
| TLS | Вручну (Traefik ACME або зовнішній) | Ingress controller + cert-manager |
| Підходить для | Хобі, малі команди, одновузлові | Команди з існуючою Kubernetes-інфраструктурою |

## Спільний контракт деплою

Незалежно від режиму деплою, застосовується наступний контракт:

- **Конфігурація**: змінні оточення через `.env.local` (Docker) або `values.yaml` (Kubernetes)
- **Секрети**: керуються оператором, ніколи не комітяться в репозиторій
- **Міграції**: явні команди, не неявні побічні ефекти запуску контейнера
- **Здоров'я**: кожен HTTP-сервіс надає `GET /health` з відповіддю HTTP 200
- **Ворота оновлення**: успіх міграцій → здоров'я core → здоров'я воркера → здоров'я публічного ентрипоінту

Дивіться [docker-deployment-contract.md](./docker-deployment-contract.md) для повного контракту.

## Локальна розробка

Локальна розробка використовує той самий Docker Compose стек що і Docker самостійний шлях.
Дивіться [docs/setup/local-dev/en/local-dev.md](../../../setup/local-dev/en/local-dev.md) для гайду з локальної розробки.

Ключова відмінність від продакшн:
- Використовуються dev-значення за замовчуванням для секретів (змінити перед виходом у продакшн)
- Вихідний код монтується як томи для live reload
- E2E-тестові контейнери запускаються паралельно з ізольованими базами даних

Назва репозиторію і частина runtime-ідентифікаторів поки що лишаються `ai-community-platform` до
окремого інфраструктурного перейменування.
