# Docker Deployment Contract

This document defines the official deployment contract for the Docker-based self-hosted path.
It covers service topology, configuration inputs, secrets, health expectations, migration behavior,
and image versioning policy.

Both Docker and Kubernetes packaging share this contract. Packaging differences do not redefine
service behavior.

## Service Topology

### Mandatory Application Services

These services must run in every supported Docker deployment:

| Service | Role | Compose file |
|---------|------|--------------|
| `traefik` | Edge proxy and routing | `compose.yaml` |
| `core` | Main platform (Symfony) | `compose.core.yaml` |
| `core-scheduler` | Background task scheduler | `compose.core.yaml` |
| `litellm` | LLM routing proxy | `compose.yaml` |

### Stateful Infrastructure (Bundled by Default)

These services are bundled in the Docker path. In Kubernetes they may be replaced by managed
external services.

| Service | Role | Compose file |
|---------|------|--------------|
| `postgres` | Primary relational database | `compose.yaml` |
| `redis` | Cache and queue backend | `compose.yaml` |
| `opensearch` | Structured log storage | `compose.yaml` |
| `rabbitmq` | Message broker | `compose.yaml` |

### Optional Platform Add-Ons

These services are included via separate compose fragments and are not required for a minimal
deployment:

| Service | Role | Compose fragment |
|---------|------|-----------------|
| `langfuse-web`, `langfuse-worker` | LLM observability | `compose.langfuse.yaml` |
| `openclaw-gateway`, `openclaw-cli` | Telegram bot gateway | `compose.openclaw.yaml` |

### Optional Agent Services

Each agent is included via its own compose fragment:

| Fragment | Agent |
|----------|-------|
| `compose.agent-knowledge.yaml` | Knowledge base agent |
| `compose.agent-hello.yaml` | Hello / greeting agent |
| `compose.agent-news-maker.yaml` | News aggregation agent |
| `compose.agent-wiki.yaml` | Wiki + grounded chat agent |
| `compose.agent-dev.yaml` | Developer assistant agent |
| `compose.agent-dev-reporter.yaml` | Pipeline reporter agent |

## Configuration Inputs

### Environment Variables

All services read configuration from environment variables. The primary input file is `.env.local`
(gitignored, operator-managed).

**Required for any deployment:**

| Variable | Description |
|----------|-------------|
| `OPENROUTER_API_KEY` | LLM provider key (or `OPENAI_API_KEY` / `ANTHROPIC_API_KEY`) |

**Required for production:**

| Variable | Description |
|----------|-------------|
| `EDGE_AUTH_JWT_SECRET` | JWT signing secret for edge authentication |
| `EDGE_AUTH_LOGIN_BASE_URL` | Public base URL of the platform (e.g. `https://yourdomain.org`) |
| `EDGE_AUTH_COOKIE_DOMAIN` | Cookie domain for cross-subdomain auth (e.g. `.yourdomain.org`) |
| `LANGFUSE_PUBLIC_URL` | Public URL for Langfuse (if Langfuse add-on is enabled) |

**Optional:**

| Variable | Description |
|----------|-------------|
| `TELEGRAM_BOT_TOKEN` | Telegram bot token (required only if OpenClaw add-on is enabled) |
| `OPENCLAW_GATEWAY_TOKEN` | Gateway auth token (auto-generated by `make bootstrap` if empty) |

### Secrets

Secrets are distributed by `make bootstrap`, which reads `.env.local` and writes:

| File | Purpose | Created by |
|------|---------|------------|
| `.env.local` | API keys and tokens | Operator (manual) |
| `docker/openclaw/.env` | OpenClaw gateway token and Telegram config | `make bootstrap` |
| `.local/openclaw/state/openclaw.json` | OpenClaw runtime config | `make bootstrap` |

**Never commit these files.** They are gitignored.

### Public URL Configuration

Production domains are configured via `compose.override.yaml` (gitignored):

```yaml
services:
  core:
    environment:
      EDGE_AUTH_LOGIN_BASE_URL: https://yourdomain.org
      EDGE_AUTH_COOKIE_DOMAIN: .yourdomain.org

  langfuse-web:
    environment:
      LANGFUSE_PUBLIC_URL: https://langfuse.yourdomain.org
      NEXTAUTH_URL: https://langfuse.yourdomain.org

  langfuse-worker:
    environment:
      LANGFUSE_PUBLIC_URL: https://langfuse.yourdomain.org
      NEXTAUTH_URL: https://langfuse.yourdomain.org
```

### Compose Override Points

The supported override mechanism is `compose.override.yaml`. This file is loaded automatically
by the Makefile `COMPOSE` variable when present. Use it to:

- Set production domain names
- Override resource limits
- Add environment-specific variables
- Disable or reconfigure optional services

Do not modify the base compose files for environment-specific settings.

## Health and Readiness

### Health Endpoints

Every HTTP service exposes `GET /health`. The expected response is HTTP 200.

| Service | Port | Health URL |
|---------|------|-----------|
| `core` | 80 | `http://localhost/health` |
| `knowledge-agent` | 8083 | `http://localhost:8083/health` |
| `news-maker-agent` | 8084 | `http://localhost:8084/health` |
| `hello-agent` | 8085 | `http://localhost:8085/health` |
| `dev-reporter-agent` | 8087 | `http://localhost:8087/health` |
| `dev-agent` | 8088 | `http://localhost:8088/health` |
| `wiki-agent` | 8090 | `http://localhost:8090/health` |

### Infrastructure Health

Stateful services expose Docker healthchecks:

| Service | Check |
|---------|-------|
| `postgres` | `pg_isready -U app -d ai_community_platform` |
| `redis` | `redis-cli ping` |
| `rabbitmq` | `rabbitmq-diagnostics -q ping` |

### Startup Dependencies

`litellm` depends on `postgres` being healthy before starting. Application services that depend
on infrastructure should use `depends_on` with `condition: service_healthy`.

## Migrations

Migrations are explicit commands, not implicit container startup side effects.

### Migration Commands

Run only the migrations for services that are installed in the environment:

```bash
make litellm-db-init       # Initialize LiteLLM database (run once, idempotent)
make migrate               # Core platform (Doctrine)
make knowledge-migrate     # Knowledge agent (Doctrine)
make dev-reporter-migrate  # Dev reporter agent (Doctrine)
make dev-agent-migrate     # Dev agent (Doctrine)
make news-migrate          # News maker agent (Alembic)
```

### Migration Order

Always run `make litellm-db-init` before starting `litellm`. Run application migrations after
the stack is up and infrastructure services are healthy.

### Migration Safety

- Migrations are idempotent and safe to re-run.
- Before upgrading, back up databases. If a migration is not backward compatible, restore from
  backup rather than rolling back the application.
- New migration targets introduced in a release are listed in the release notes.

## Image Versioning and Upgrade Policy

### Current Model (Source-Based)

The current deployment model builds images from source on the target host:

- `make up` runs `docker compose ... up --build -d`
- The deployed revision is the checked-out Git commit
- Rollback is `git checkout <previous-ref>` followed by `make up`

### Future Model (Published Images)

When the platform publishes versioned images, the upgrade flow will evolve to:

1. Update image tags in `compose.override.yaml` or the supported compose inputs
2. Run `docker compose pull`
3. Run explicit migration commands
4. Restart services in supported order
5. Verify health and roll back to previous pinned tags if required

### Compatibility Matrix

| Component | Compatibility notes |
|-----------|-------------------|
| `core` | Requires matching `postgres` schema version |
| `knowledge-agent` | Requires matching `knowledge_agent` schema version |
| `news-maker-agent` | Requires matching `news_maker_agent` schema version |
| `dev-reporter-agent` | Requires matching `dev_reporter_agent` schema version |
| `litellm` | Requires `litellm` database to exist (run `make litellm-db-init`) |
| `openclaw-gateway` | Pinned to a specific image digest in `compose.openclaw.yaml` |

Always review release notes before upgrading. Breaking schema changes require backup before
migration.

## Supported Compose Bundle

The full supported stack is assembled by the Makefile `COMPOSE` variable:

```
compose.yaml                    # Infrastructure + LiteLLM
compose.core.yaml               # Core platform + scheduler
compose.agent-*.yaml            # Optional agents (all present files are included)
compose.langfuse.yaml           # Optional: LLM observability
compose.openclaw.yaml           # Optional: Telegram bot gateway
compose.override.yaml           # Operator overrides (if present)
```

Use `make ps` to see the running services for the current compose bundle.
